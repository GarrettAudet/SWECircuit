# Milestone: V11 Portable Orchestration Control Plane

## Status

Architecture revision 2 in review preparation. Round 1 returned `REVISE`; no V11 runtime capability is implemented, technically accepted, merge-ready, or merged.

## Goal

Give a developer one IDE-agnostic workflow that takes a software goal through user-defined Modules and a Circuit, bounded decomposition, specialized capability assignment, safe parallel execution, integrated verification and review, owner approval, a full execution trace, and evidence-backed learning.

- Revision 2 resolves the findings in the proposed architecture and test obligations.
- Runtime contracts, code, fixtures, dogfood, final review, and acceptance remain ahead.

## Why It Matters

This pre-implementation gate is the first real dogfood of the intended multi-agent system. It decomposed one architecture goal into four bounded specialist reviews, ran them without write conflicts, bound every handoff to one commit, and stopped implementation when fan-in found material defects. Revision 2 turns those findings into closed contracts and executable tests before code makes them expensive.

## Revision-2 Design

- One `runGoal` facade defaults to one agent and concurrency one.
- Many agents use the same contract with more allowlisted profiles and bounded slots.
- Circuit owns workflow policy; planner data can only instantiate permitted work.
- RunAuthority roots permissions independently of planner, profile, and manifest claims.
- Plans are immutable; capability assignments bind live availability in state.
- One serialized coordinator claims a complete conflict-safe wave before callbacks.
- Disjoint work may run concurrently; overlapping or unknown writes serialize.
- Every child result is bound to its parent, packet, worker, commit, ports, journal, and evidence.
- Failure, diagnosis, cancellation, uncertainty, joins, integration, verification, review, approval, and memory candidates have explicit routes.
- Core does not select APIs/models/providers/IDEs, merge branches, retry automatically, or mutate memory.

## Residual Risks

- Revision 2 may still contain an implementation-level ambiguity found by Round 2.
- Complete waves can wait for the slowest worker.
- Capability profiles are host-attested contracts, not proof of worker quality.
- Conservative scope rules may serialize useful work.
- V10 remains an owner-gated stacked baseline.
- Public reuse remains legally unclear until a license is selected.
